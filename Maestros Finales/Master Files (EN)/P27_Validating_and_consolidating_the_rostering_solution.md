## Confirming the solution is ready to validate

After reviewing coverage, conflicts, and feasibility, the next step is deciding whether the Rostering solution is solid enough. Validating is not just an administrative approval—it is declaring the staff assignment is coherent, understandable, and usable as an approved baseline.

Use this quick start when you already ran Rostering calculation, analyzed the result, and need to formally close the solution before consolidating it.

Before you begin, make sure that:
1. You ran Rostering calculation in P25.
2. You reviewed conflicts, coverage, and feasibility in P26.
3. You corrected major issues or you understand why remaining conflicts are acceptable.
4. You know which exact solution you will validate.

For this quick start, use this reference case:

> **I’m going to validate the Rostering solution for line L1 because assignment covers the work reliably enough and I want to consolidate it as an approved reference.**

To confirm readiness:
1. Open the Rostering solution you will use as reference.
2. Review work coverage one last time.
3. Confirm the main conflicts are resolved or clearly diagnosed.
4. Confirm the resulting assignment is coherent with:
   1. Rostering rules,
   2. real staff availability,
   3. operational assignment,
   4. and the solution inherited from Scheduling.
5. If you detect an unresolved major issue, do not validate yet.
6. If the baseline is stable, continue to validation.

For the reference case, do not proceed until you can state:
1. L1 work is covered or remaining gaps are understood.
2. The solution is operationally defensible.
3. You do not need structural changes before approval.

When you finish this section, you should know whether the solution deserves formal validation.

## Validating the staff solution

Once the solution is stable enough, run validation. This step closes calculation and review and turns the solution into an approved reference in the workflow.

Before you start this section, make sure that:
1. You decided the solution is valid.
2. You no longer need to recalculate or adjust rules before approval.
3. You understand validation freezes the solution as an approved baseline.

To validate the Rostering solution:
1. From the solution view or the main table, open the corresponding actions menu.
ref: P27_Imagen1.png | full
2. Select **Validate**.
3. Review the solution summary one last time.
4. Confirm validation when prompted.
5. Confirm solution status changes to the corresponding approved status.
6. Confirm the solution is no longer treated as provisional work.
7. If your flow uses specific approval permissions, confirm validation was recorded properly.

For the reference case, make sure that:
1. L1 solution changes status after validation.
2. The system recognizes it as an approved version.
3. The solution stops being treated as an open iteration.

When you finish this section, you should have a formally validated Rostering solution.

## Consolidating the solution as an operational reference

After validation, consolidate the solution. Consolidation means treating this version as the approved reference for the next process level. From here, it should no longer be treated as a test, but as the serious, traceable baseline for staff assignment.

Before you start this section, make sure that:
1. The solution is validated.
2. You know whether this will be the active reference or an approved version pending later use.
3. You can distinguish an approved solution from a solution still under review.

To consolidate the validated solution:
1. Review the solution name and description.
2. If needed, update the description to make clear:
   1. what context it covers,
   2. what period it represents,
   3. why it was approved.
3. Confirm the validated solution is clearly distinguishable from drafts, tests, or prior iterations.
4. If your internal process requires it, record that this version becomes the reference for the next step.
5. Keep prior versions as history, but avoid treating them as equivalent to the approved solution.

For the reference case, make sure that:
1. The validated L1 solution is clearly distinguished from intermediate versions.
2. The team can identify it as the correct reference.
3. Approval traceability is clear.

When you finish this section, you should have an approved solution that is recognizable as the Rostering reference.

## Understanding what is locked and what requires a new iteration

Before closing, be clear that validating does not remove the possibility of improvement—it closes that specific version. If you later need significant improvements, the right approach is creating a new iteration (or a new working solution), not modifying the approved version without control.

Before you continue, make sure that:
1. The solution is validated.
2. You know which parts are now closed.
3. You understand future improvements must be traced as new iterations.

To keep post-validation governance clear:
1. Treat the validated solution as a closed reference.
2. Avoid modifying it directly as if it were still a draft.
3. If you detect a future improvement:
   1. create a new iteration,
   2. or start a new calculation/review cycle.
4. Keep the validated solution as the historical comparison baseline.
5. If your team audits decisions, use this solution as the approved reference.

For the reference case, finish this section only when you can state:
1. L1’s validated version is closed.
2. Any future improvement will be done via a new iteration.
3. Traceability across calculation, review, and approval is preserved.

When you finish this section, you should understand what it means to consolidate a solution and how to keep version control.

## Leaving the solution ready for the next process level

The last step is preparing the transition. From here, the Rostering solution is no longer in technical calculation—it is in use, consolidation, or transfer to the next operational process as needed.

Before you finish, make sure that:
1. You validated the solution.
2. You treated it as a consolidated reference.
3. You know whether the next step is:
   1. communicating it,
   2. integrating it,
   3. auditing it,
   4. or preparing a future iteration.

To close this quick start:
1. Review the solution status one last time.
2. Confirm it is no longer a provisional calculation.
3. Confirm the team could identify it as the approved version.
4. If your process requires it, record the transition to the next operational level.
5. Keep the solution as a stable reference for future comparison.

For the reference case, finish this quick start only when you can state:
1. L1’s Rostering solution is validated.
2. It is consolidated as an approved reference.
3. The next step is no longer calculating, but using/reviewing/evolving the baseline in a controlled way.

When you finish this section, you should have a validated, consolidated Rostering solution ready to serve as a stable process reference.
